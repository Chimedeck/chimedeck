# Changelog — 20260323_113807

## Update
- Mounted `boardTypePreferencesRouter` in `server/extensions/notifications/api/index.ts` under `/api/v1/boards/:boardId/notification-preferences/types` (takes precedence over the existing `/notification-preference` route to avoid prefix collision).

## New
- `server/extensions/notifications/api/boardTypePreferences/get.ts` — GET handler that returns all 9 notification types with resolved values (`in_app_enabled`, `email_enabled`, `source: 'board' | 'user' | 'default'`) by merging board-level overrides → user-level prefs → opt-out default.
- `server/extensions/notifications/api/boardTypePreferences/update.ts` — PATCH handler that upserts a single `board_notification_type_preferences` row and returns the refreshed full list (same shape as GET).
- `server/extensions/notifications/api/boardTypePreferences/reset.ts` — DELETE handler that removes all `board_notification_type_preferences` rows for the current user+board.
- `server/extensions/notifications/api/boardTypePreferences/index.ts` — Router that matches `/api/v1/boards/:boardId/notification-preferences/types` and delegates to the three handlers.
- `tests/integration/notifications/boardTypePreferences.test.ts` — Auth/validation integration tests for all three endpoints (6 tests, all passing).
