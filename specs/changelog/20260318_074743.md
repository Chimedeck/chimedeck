# Changelog — 2026-03-18

## Update
- `server/middlewares/boardVisibility.ts`: Extended `BoardVisibilityScopedRequest` interface with an optional `guestType` field. When the authenticated caller is a GUEST, the middleware now reads `guest_type` from `board_guest_access` and attaches it as `req.guestType` (defaults to `VIEWER` for backward compatibility). Downstream write-gate handlers use `guestCanWrite(req.guestType)`.
- `server/extensions/board/api/guests/create.ts` (email-based invite): Accepts optional `guestType` in the POST body, validates it, defaults to `VIEWER`, and persists it as `guest_type` on `board_guest_access`. Response now returns `guestType`, `grantedAt`, and `grantedBy` in camelCase.
- `server/extensions/board/api/guests/index.ts`:
  - Legacy `handleInviteGuestById`: Same `guestType` acceptance and persistence as the email-based handler.
  - `handleListGuests`: Now selects `guest_type as guestType`, `granted_at as grantedAt`, `granted_by as grantedBy` so the field is included in the response.
  - Exports `handleUpdateGuestType` from the new module.
- `server/extensions/board/api/index.ts` (board router): The guest user sub-route (`/guests/:userId`) now dispatches `PATCH` to `handleUpdateGuestType` in addition to `DELETE` for `handleRevokeGuest`.

## New
- `server/extensions/board/api/guests/updateGuest.ts`: New `PATCH /api/v1/boards/:id/guests/:userId` endpoint. Validates `guestType` (VIEWER | MEMBER), requires ADMIN+ role, returns 404 if the guest row does not exist, updates `guest_type` in `board_guest_access`, and emits a `member_updated` event.
- `specs/tests/sprint-89-guest-role-split.md`: Appended Scenarios 18–27 covering middleware `guestType` attachment (including null fallback), invite endpoint with `guestType`, PATCH endpoint (happy path, 404, 400, 403), and list endpoint `guestType` field.

## Technical Debt
- The legacy `handleInviteGuestById` function in `guests/index.ts` remains for internal compatibility but is not exposed via any public route; it should be removed or promoted to a named export once the email-based flow is the sole invite path.
