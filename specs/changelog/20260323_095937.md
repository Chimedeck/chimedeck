# Changelog — 20260323_095937

## Update

- `server/extensions/mcp/http/sessions.ts` — Updated `McpSession.transport` type from `StreamableHTTPServerTransport` (Node.js) to `WebStandardStreamableHTTPServerTransport` (web-standard, correct for Bun).

## New

- `server/extensions/mcp/http/index.ts` — `mcpHttpHandler` implementation:
  - Handles `POST /api/mcp` (no `mcp-session-id`): initializes a new MCP session, creates `McpServer` + `WebStandardStreamableHTTPServerTransport`, registers all 6 tools, connects and returns the MCP initialize response with `mcp-session-id` header.
  - Handles `POST /api/mcp` (with `mcp-session-id`): routes tool calls/notifications to the existing session's transport.
  - Handles `GET /api/mcp`: opens SSE stream for the session.
  - Handles `DELETE /api/mcp`: closes transport and evicts session (returns 204).
  - Authentication enforced at the entry point — no MCP logic runs before identity is confirmed.
  - Session ownership check (userId match) on every subsequent request to prevent hijacking.
  - Returns `null` for non-`/api/mcp` paths so it is safe to compose with other routers.
- `server/index.ts` — Imported `mcpHttpHandler` and wired it before the 404 fallback.
- `specs/tests/sprint-106-mcp-http-init.md` — Playwright MCP test spec covering: unauthenticated POST (401), valid POST returning `mcp-session-id` (200), DELETE terminating session (204/404), missing session (400/404), and session hijacking prevention (403).

## Technical Debt

- The `onsessioninitialized` callback is used to register the session into the Map. If `WebStandardStreamableHTTPServerTransport` does not call the callback synchronously before `handleRequest` resolves (e.g. due to async initialization), the session could briefly not be in the Map. Future improvement: investigate whether eager insertion before `handleRequest` is safer.
- In-memory sessions are lost on server restart. A future sprint could add Redis-backed session storage.
