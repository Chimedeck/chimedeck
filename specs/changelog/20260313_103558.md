# Changelog — Sprint 78, Iteration 1: board_members Migration + Board Creation Auto-Insert

## Update

- `server/extensions/board/api/create.ts` — board creation is now wrapped in a DB transaction. After inserting the board, the creator is automatically inserted into `board_members` with `role = 'ADMIN'`. The `actorId` in the event emission now consistently uses the resolved creator ID instead of a fallback to `'system'`.

## New

- `db/migrations/0040_board_members.ts` — new migration that creates the `board_members` table with columns: `id`, `board_id` (FK → boards), `user_id` (FK → users), `role` (ADMIN | MEMBER), `created_at`, `updated_at`. Includes a composite unique constraint on `(board_id, user_id)` and indexes on both FK columns.
- `specs/tests/sprint-78-iteration-1-board-members-migration.md` — Playwright MCP test walkthrough that creates a board and verifies the creator appears as `ADMIN` in `board_members`.

## Technical Debt

- The `GET /api/v1/boards/:id/members` endpoint referenced in the test does not yet exist; it will be implemented in Iteration 3. Until then the acceptance criteria can only be verified via direct DB inspection.

## What Should Be Done Next

- Iteration 2: Implement `boardVisibility` middleware to enforce the PRIVATE / WORKSPACE / PUBLIC access matrix on board-scoped routes.
