# Changelog — 2026-03-12T02:06:06Z

## Update
- `server/extensions/automation/config/index.ts`: Added `usePgCron` and `schedulerEnabled` flags driven by `AUTOMATION_USE_PGCRON` and `AUTOMATION_SCHEDULER_ENABLED` env vars.
- `server/extensions/automation/engine/index.ts`: Added `execute()` function for direct scheduler invocation (SCHEDULED/DUE_DATE automations by ID, used by listener and Worker fallback).
- `server/index.ts`: Wired `startAutomationScheduler()` into server startup sequence.
- `.env.example`: Added `AUTOMATION_SCHEDULER_ENABLED` and `AUTOMATION_USE_PGCRON` with documentation.

## New
- `server/extensions/automation/scheduler/listener.ts`: Bun-native pg LISTEN client for the `automation_tick` pg_notify channel. Reconnects automatically (5 s backoff) on PostgreSQL restarts. Used when `AUTOMATION_USE_PGCRON=true`.
- `server/extensions/automation/scheduler/workerFallback.ts`: Bun Worker thread that calls `automation_scheduler_tick()` via SQL every 60 s. Used in local dev when `AUTOMATION_USE_PGCRON=false` (postgres:16-alpine lacks pg_cron).
- `server/extensions/automation/scheduler/index.ts`: Entry point that starts either the LISTEN client or Worker fallback based on `automationConfig.usePgCron`. No-ops when `schedulerEnabled=false`.
- `tests/integration/automation/scheduler.test.ts`: 9 integration tests verifying config flags, scheduler export, execute() no-op on missing automation, listener/worker module contracts.

## Technical Debt
- The `execute()` function constructs a synthetic `AutomationEvent` for SCHEDULED/DUE_DATE runs; a dedicated `SchedulerEvent` type would be cleaner but is deferred.
- `Worker` is cast via `as any` to work around missing `Worker` global type in the bun tsconfig; a proper `@types/bun` update would resolve this.
