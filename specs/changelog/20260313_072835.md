# Changelog — 2026-03-13T07:28:35Z

## Update

- `src/slices/featureFlagsSlice.ts` — Added `notificationPreferencesEnabled` and `emailNotificationsEnabled` flags to the client state, fetch response shape, reducer, and exported selectors (`selectNotificationPreferencesEnabled`, `selectEmailNotificationsEnabled`).
- `server/index.ts` — `GET /api/v1/flags` now resolves and returns `notificationPreferencesEnabled` and `emailNotificationsEnabled` from the flags service.
- `src/extensions/UserProfile/containers/EditProfilePage/EditProfilePage.tsx` — Mounts `<NotificationPreferencesPanel />` below the ChangeEmailForm section, wrapped in a `<section>` with an "h2 Notification Preferences" heading. Gated by `selectNotificationPreferencesEnabled` — section not rendered when flag is off.

## New

- `src/extensions/Notifications/NotificationPreferences/NotificationPreferencesPanel.tsx` — Table-of-toggles component. Renders one row per notification type (4 types) with In-App and Email columns. Uses native `button[role="switch"]` accessible toggles styled in Tailwind. Calls `useGetNotificationPreferencesQuery` on mount; each toggle fires `useUpdateNotificationPreferencesMutation`. Opts-out model: missing preference rows render as enabled by default. Email toggles are disabled with a tooltip when `SES_ENABLED=false` or `EMAIL_NOTIFICATIONS_ENABLED=false`. API errors surface as an error toast via local `useState` + `ToastRegion`.
- `specs/tests/notification-preferences-ui.md` — Playwright MCP test spec with 9 scenarios covering: panel renders 4 types, default opt-out state, toggle fires PATCH with optimistic update, API error rollback with error toast, email toggles greyed when SES disabled, email toggles active when SES enabled, panel hidden when flag is off, aria-label accessibility, and loading state.
