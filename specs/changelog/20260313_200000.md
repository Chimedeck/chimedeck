# Changelog — 2026-03-13T19:50:00Z

Sprint 80 — Iteration 9: Guest Client Permission Guards + Auth-Aware Route Guards

## Update

- `src/extensions/Board/components/BoardHeader.tsx`: Added `isGuest?: boolean` prop. When `true`, the member avatar stack and the `···` settings menu are hidden from the header. The automation button remains visible (read-only for guests).
- `src/extensions/Board/components/BoardSettings/index.tsx`: Added `isGuest?: boolean` prop. Returns `null` immediately when `isGuest` is true, preventing GUEST users from viewing or interacting with the board settings panel.
- `src/extensions/Board/components/BoardMembersPanel/index.tsx`: Added `isGuest?: boolean` prop. Returns `null` immediately when `isGuest` is true, preventing GUEST users from viewing or interacting with the board members panel.
- `src/extensions/Board/containers/BoardSettings/BoardSettings.tsx`: Added `isGuest?: boolean` prop (mirrors the component counterpart). Returns `null` when `isGuest` is true. This is the file actually imported by BoardPage.
- `src/extensions/Board/containers/BoardPage/BoardPage.tsx`: Imports `selectIsGuestInActiveWorkspace` from `workspaceSlice`. Derives `isGuest` via `useAppSelector` and passes it to `BoardHeader`, `BoardSettings`, and `BoardMembersPanel`. Uses object spreading to pass optional board action handlers only for non-guest users, satisfying `exactOptionalPropertyTypes`.
- `src/routing/index.tsx`: Wraps `/workspace/:workspaceId` route with the new `GuestBlockedRoute` component so GUEST users are redirected at the router level.

## New

- `src/routing/GuestBlockedRoute.tsx`: Route-level guard that redirects workspace GUEST users away from the workspace members/settings page (`/workspace/:workspaceId`). Redirects to `/workspaces/:workspaceId/boards` when a workspace ID is available, otherwise to `/workspaces`. Uses `selectIsGuestInActiveWorkspace` for the role check.
- `specs/tests/sprint-80-iteration-9-guest-permissions.md`: Playwright MCP test walkthrough covering: GUEST cannot see board header settings menu or avatar stack; GUEST can create/edit/delete cards; GUEST is redirected from the workspace members route; regular admin still sees all controls (regression check).

## Technical Debt

- `containers/BoardSettings/BoardSettings.tsx` and `components/BoardSettings/index.tsx` are duplicate files with overlapping responsibilities. The container variant is what BoardPage actually uses; the component variant appears to be an extended version with role-aware visibility selector and boardMembersSlice integration. These should be consolidated into one file.
- `GuestBlockedRoute` depends on the Redux store having the active workspace's `callerRole` populated. If the user deep-links to `/workspace/:workspaceId` before the workspace list has loaded, the guard will not fire (returns `<Outlet />`). The WorkspacePage already shows an "Access Restricted" notice as a secondary guard, providing a fallback.

## What Should Be Done Next

- Remove the now-redundant `/workspace/:workspaceId/boards` route from `src/extensions/Board/routes.ts` as noted in the previous iteration's changelog.
- Add E2E tests that exercise the full guest invite → board access → permission enforcement flow.
