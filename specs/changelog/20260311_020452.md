# Changelog — 2026-03-11T02:04:52Z

## Update

- `src/extensions/Realtime/client/messageQueue.ts` — Updated `enqueue()` to fire-and-forget persist to IndexedDB via `enqueueMutation()`, and `dequeue()` to fire-and-forget remove via `acknowledgeMutation()`. Added `hydrate()` method for boot-time population from persisted data.
- `src/App.tsx` — Wired `loadPersistedMutations()` into the app boot `useEffect` so pending mutations from IndexedDB are loaded into the in-memory queue before the first WS connection attempt.
- `package.json` / `bun.lock` — Added `idb@8.0.3` dependency.

## New

- `src/mods/offlineQueue.ts` — IndexedDB-backed persistence layer for pending mutations. Exports three functions:
  - `loadPersistedMutations()` — reads all pending mutations from IDB sorted by `enqueuedAt`, called once on app boot.
  - `enqueueMutation(mutation)` — persists a `PendingMutation` to IDB when it is enqueued.
  - `acknowledgeMutation(id)` — removes a mutation from IDB once it has been successfully replayed.
  - Errors are caught and logged as warnings so IndexedDB unavailability (private browsing, storage quota) degrades silently.
- `tests/e2e/offline-queue-persistence.spec.ts` — Two Playwright E2E tests:
  1. Verifies a mutation seeded in IndexedDB before a hard reload is still present after reload (proves persistence survives page refresh).
  2. Verifies the mutation is removed from IndexedDB after the WS reconnects and the replay HTTP call is made (proves `acknowledgeMutation` is called on successful dequeue).

## Technical Debt

- The `messageQueue.clear()` method (used on overflow) does not clear IndexedDB. If overflow occurs the in-memory queue is cleared but the persisted entries remain. A follow-up should call `clearAllMutations()` in IDB on overflow.
- The `hydrate()` boot flow loads mutations before the WS is open; if replay happens at WS connect time only (current behaviour), there is a window where the hydrated mutations are in memory but not yet sent. This is by design and matches spec §6, but worth noting.

## What Should Be Done Next

- Consider exposing a test hook or window-level observable so E2E tests can assert queue state without polling IndexedDB directly.
