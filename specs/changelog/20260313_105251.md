## Update
- Updated `server/middlewares/boardVisibility.ts` to implement the full access matrix:
  - PRIVATE boards: workspace OWNER/ADMIN bypass the board_members check; MEMBER/VIEWER require an explicit `board_members` row; GUEST requires a `board_guest_access` row; unauthenticated requests get 403.
  - WORKSPACE boards: all workspace members are allowed; GUESTs are denied (403); unauthenticated requests get 403.
  - PUBLIC boards: no authentication required for reads (unchanged).
- Updated `server/extensions/board/api/index.ts` to apply `applyBoardVisibility` at the router level for all `/api/v1/boards/:id[/...]` routes before dispatching to individual handlers.
- Updated `server/extensions/list/api/index.ts` to apply board visibility at the router level for both board-scoped (`/api/v1/boards/:boardId/lists`) and list-scoped (`/api/v1/lists/:id`) routes.
- Updated `server/extensions/card/api/index.ts` to apply board visibility at the router level for list-scoped card routes, checklist item routes, and all card-scoped routes.

## New
- Added `applyBoardVisibilityFromList` helper to `boardVisibility.ts` — looks up the board from a list ID, then applies visibility enforcement.
- Added `applyBoardVisibilityFromCard` helper to `boardVisibility.ts` — looks up the board from a card ID (via its list), then applies visibility enforcement.
- Added `applyBoardVisibilityFromChecklistItem` helper to `boardVisibility.ts` — looks up the board from a checklist item ID (via card → list → board), then applies visibility enforcement.
- Added `specs/tests/sprint-78-iteration-2-board-visibility.md` Playwright MCP test covering PRIVATE/WORKSPACE/PUBLIC access scenarios.

## Technical Debt
- Individual board handlers that already call `applyBoardVisibility` themselves (e.g. `get.ts`, `labels.ts`, `members.ts`, `comments.ts`, `archived-cards.ts`) now run the visibility check twice — once at the router level and once inside the handler. This results in redundant DB queries and auth checks. A follow-up refactor should remove the per-handler calls once confidence in the router-level guard is established.
- The checklist item PATCH/DELETE handlers also do their own workspace membership check internally, which is now redundant with the router-level visibility guard.
