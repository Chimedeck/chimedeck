# Changelog — 20260310_044855

## Update
- `server/extensions/auth/common/config/jwt.ts`: `accessTokenTtlSeconds` now reads from `env.ACCESS_TOKEN_TTL_SECONDS` (was hardcoded `15 * 60`). Default is **24 hours** (86400 s) — overridable via `ACCESS_TOKEN_TTL_SECONDS` env var.
- `server/extensions/auth/api/logout.ts`: After revoking the refresh token the handler now fetches the `user_id` from the token row and publishes `{ type: "session_revoked" }` to `session:<userId>` on the pub/sub bus.
- `server/extensions/auth/api/resetPassword.ts`: After deleting all refresh tokens for the user also publishes `{ type: "session_revoked" }` to `session:<userId>`.
- `server/extensions/realtime/api/index.ts`: `open()` now calls `subscribeSessionRevocation(userId)` to begin listening for logout events; `close()` calls `unsubscribeSessionRevocation(userId)` to clean up when the last socket for a user disconnects.

## New
- `server/config/env.ts`: Added `ACCESS_TOKEN_TTL_SECONDS` config entry (default `24 * 60 * 60 = 86400`).
- `server/extensions/realtime/sessionRevocation.ts`: New helper that subscribes to the `session:<userId>` pub/sub channel; when `session_revoked` is received it closes all open WebSocket connections for that user with **close code 4001** (`session revoked`), enabling the client to redirect to `/login?reason=session_revoked`.

## Technical Debt
- `unsubscribeSessionRevocation` calls `pubsub.unsubscribe(channel)` which removes **all** listeners on that channel (InMemoryPubSubAdapter uses `removeAllListeners`). If the Redis adapter ever registers additional listeners on the same channel this could silently drop them. The adapters should be extended to support per-handler unsubscription.

## What Should Be Done Next
- Iteration 6: Client-side forced logout (401 interceptor in RTK Query), WS close code 4001 → redirect, HTTP polling fallback (`GET /api/v1/boards/:id/events?since=<lastSequence>`), and server endpoint for event polling.
