# Changelog — 2026-03-17T05:28:03Z

## Sprint 83 Iteration 8 — Idempotent Comment Create + Card Description Patch Offline Replay

### Update

- **`server/extensions/comment/api/create.ts`** — accepts optional `idempotency_key` in the POST body. When present, the handler checks for an existing comment belonging to the same `user_id` with that key before inserting. If found, the existing comment is returned (`201`) without side-effects, preventing duplicate comments during offline queue replay.
- **`server/extensions/card/api/patch.ts`** — rewritten as a description-specific endpoint (`PATCH /api/v1/cards/:id/description`). Accepts `description`, `idempotency_key`, and `client_updated_at` replay metadata. If both `idempotency_key` and `client_updated_at` are supplied and the card's server-side `updated_at` is already ≥ `client_updated_at`, the handler returns the current card without re-applying the update or emitting duplicate activity/WS events.
- **`server/extensions/card/api/index.ts`** — wires `PATCH /api/v1/cards/:id/description` to `handlePatchCardDescription`.

### New

- **`db/migrations/0084_comment_idempotency_key.ts`** — adds nullable `idempotency_key` column to `comments` table with a unique constraint scoped per `(user_id, idempotency_key)`.
- **`specs/tests/offline-replay-save-and-comment.md`** — 16 Playwright MCP scenarios covering idempotent comment creation (per-user key scoping, duplicate detection, empty-key rejection) and idempotent description PATCH (replay detection via `client_updated_at`, invalid-timestamp rejection, missing-description rejection) plus end-to-end offline/online transition and hard-refresh recovery flows.

### Technical Debt

- The description idempotency check uses `client_updated_at` vs `card.updated_at` as a heuristic. This is a pragmatic approach that covers the common offline replay case, but cannot distinguish between two _different_ edits with the same timestamp. A dedicated `idempotency_keys` table (keyed by `user_id + resource_type + idempotency_key`) would give stronger guarantees at the cost of an additional lookup.
