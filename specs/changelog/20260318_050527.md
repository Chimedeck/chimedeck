## Update
- Modified `server/extensions/card/api/move.ts` to emit a `card_moved` activity event after a successful cross-list move. Same-list reorders are intentionally excluded via the `emitCardMoved` guard (`fromListId === toListId`). Both `dispatchEvent` and `emitCardMoved` are now awaited together in `Promise.all` to avoid latency stacking.
- Modified `server/extensions/card/api/members.ts` to emit `card_member_assigned` on first-time assignment (idempotent repeat returns 200 without emitting), and `card_member_unassigned` only when the membership row actually existed before the delete (prevents phantom unassign events). Refactored `resolveWorkspaceId` → `resolveCardContext` to return `boardId` alongside `workspaceId` so activity events can be fully populated without extra DB round-trips.
- Updated `server/extensions/activity/config/visibleEventTypes.ts` to include `'card_moved'`, `'card_member_assigned'`, and `'card_member_unassigned'` in the visible event type allowlist.

## New
- Extended `server/extensions/activity/mods/createActivityEvent.ts` with three new typed emitter functions: `emitCardMoved` (with no-op guard for same-list reorders), `emitCardMemberAssigned`, and `emitCardMemberUnassigned`. Each enforces the payload contract at the call site.
- Created `specs/tests/card-activity-create-move-assign.md` — full Playwright MCP scenario specification covering `card_created`, `card_moved`, `card_member_assigned`, and `card_member_unassigned` event emission, including idempotency, error paths, combined sequences, and schema conformance assertions.

## Technical Debt
- `members.ts` now performs an extra `SELECT` before each `DELETE` (to check existence for idempotency of the unassign event). This is acceptable at current load but could be replaced with a `DELETE … RETURNING` pattern in a future refactor.
- Test spec remains in requirements/Markdown format; needs conversion to executable Playwright test file.
