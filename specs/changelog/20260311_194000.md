# Changelog — 2026-03-11T19:40:00Z

## Update
- `db/migrations/0035_automation_scheduler.ts`: New DB migration for Sprint 64 automation scheduler SQL functions.

## New
- **`automation_should_run_now(automation_id uuid) → boolean`**: Pure SQL function that reads `automations.config` (scheduleType, hour, minute, dayOfWeek/dayOfMonth/month) and compares against `NOW()` UTC to determine if a SCHEDULED automation should fire this minute.
- **`automation_due_date_in_window(due_date timestamptz, trigger_config jsonb) → boolean`**: Pure SQL function that applies the `offsetDays`/`triggerMoment` config to shift a card's due_date and checks whether `NOW()` falls within the ±30s window around the computed target timestamp.
- **`automation_scheduler_tick() → void`**: Master tick procedure (called every minute by `pg_cron` in production or the Bun Worker fallback locally). Issues `pg_notify('automation_tick', ...)` for each eligible SCHEDULED or DUE_DATE automation, and atomically updates `last_run_at` on SCHEDULED automations to prevent double-fires.

## Technical Debt
- `CREATE EXTENSION pg_cron` and `cron.schedule()` are intentionally excluded from the migration (require superuser). These must be run by the DBA as a one-time pre-deploy step on production (see `specs/sprints/sprint-64.md §2`).
