# Changelog — 2026-03-18T05:25:40Z — Sprint 88 Iteration 9: Realtime activity updates in open card modal

## Update

- `server/extensions/activity/mods/createActivityEvent.ts`: All four `emit*` helpers (`emitCardCreated`, `emitCardMoved`, `emitCardMemberAssigned`, `emitCardMemberUnassigned`) now call `publishCardActivityEvent` after writing the activity record. The call is fire-and-forget so a WS delivery failure never blocks the API response.
- `src/extensions/Card/slices/cardDetailSlice.ts`: Added `addActivity` reducer that appends an incoming realtime `ActivityData` row only when its `entity_id` matches `state.openCardId`, providing both card-scope filtering and deduplication in one place.
- `src/extensions/Realtime/hooks/useBoardSync.ts`: Added `card_activity_created` case to the event switch; dispatches `cardDetailSliceActions.addActivity` so all board-room subscribers receive live activity updates without a page reload.

## New

- `server/extensions/activity/events/publishCardActivityEvent.ts`: New module that resolves actor display info (name, email, resolved avatar URL) from the DB and broadcasts a `card_activity_created` event to the board room via the existing `publisher` infrastructure.
- `specs/tests/card-activity-create-move-assign.md` Section 9: Seven Playwright MCP multi-session scenarios covering:
  - New activity row appears in open modal without refresh (9.1)
  - Self-action realtime row for the actor (9.2)
  - Assignment event received by a concurrent session (9.3)
  - Cross-card isolation — events for a different card are ignored (9.4)
  - Realtime delivery after WebSocket reconnect (9.5)
  - No cross-board leakage (9.6)
  - Concurrent comment + realtime activity — no duplicate or missing rows (9.7)

## Technical Debt

- Avatar URL resolution in `publishCardActivityEvent` issues one extra DB query per activity write. For high-throughput boards a cache (e.g. per-request map keyed by actorId) would reduce load. Deferred to a future performance iteration.
- The `card_activity_created` realtime event currently reaches every board-room subscriber regardless of whether they have that card modal open; the reducer silently discards events for non-open cards. A card-scoped subscription channel would be more efficient but is out of scope for this iteration.
