# Sprint 132 — Comment Threaded Replies UI

## Update

- Extended `CommentData` interface in `cardDetail.ts` with `parent_id` and `reply_count` fields.
- Extended `Comment` interface in `CommentItem.tsx` with `parent_id` and `reply_count`.
- Added `Reply` button to `CommentItem` action bar (top-level comments only — `parent_id` is null).
- `CommentItem` now renders a `CommentReplyThread` below each top-level comment that has replies or an open reply editor.

## New

- `src/extensions/Comment/api.ts` — added `getReplies` (GET `/comments/:id/replies`) and `postReply` (POST `/cards/:id/comments` with `parent_id`).
- `src/extensions/Comment/components/CommentReplyThread.tsx` — new component. Manages load-on-demand replies with expand/collapse toggle and an inline reply `CommentEditor`. Enforces max depth = 1 by not passing `onAddReply` to child `CommentItem` instances.
- `src/extensions/Comment/translations/en.json` — added `comment.action.reply`, `comment.replies.viewOne`, `comment.replies.viewMany`, `comment.replies.hide`, `comment.reply.placeholder`, `comment.reply.submit`.

## Technical Debt

- `CommentReplyThread` maintains replies in local state rather than in Redux. This means real-time WS events (`comment_reply_added`) only increment the parent's `reply_count` in Redux; the reply list itself refreshes via `loadReplies()` after each new reply. A future iteration could store replies in Redux keyed by `parentId` for true real-time thread sync.

## What Should Be Done Next

- Wire `onEditReply` / `onDeleteReply` through from the container (currently stubs are passed).
- Store replies in Redux slice keyed by parentId so real-time `comment_reply_added` events can update the open thread without a re-fetch.
- Add Playwright E2E tests per the sprint spec: reply flow, counter persistence on reload, real-time sync across two tabs.
