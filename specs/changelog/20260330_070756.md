# Changelog — 2026-03-30T07:07:56Z

## Update

- **server/extensions/realtime/userChannel.ts** — `publishToUser` is now `async` and routes messages through `pubsub.publish('user:<userId>', ...)` instead of iterating `userSockets` directly. Added `subscribeUserChannel` and `unsubscribeUserChannel` functions mirroring the pattern in `sessionRevocation.ts`. The pubsub subscriber delivers the message to all locally-connected sockets for the user on the receiving instance.
- **server/mods/pubsub/adapters/redis.ts** — Added duplicate-subscribe guard to `subscribe()`: throws `Error('Already subscribed to channel: <channel>')` if the channel is already registered in `this.handlers`. Prevents double-delivery and ambiguous unsubscribe semantics.
- **server/mods/pubsub/adapters/redis.test.ts** — Added test for the duplicate-subscribe guard (mocks the internal `sub` Redis client to avoid needing a live connection).

## New

- `subscribeUserChannel(userId)` — subscribe this server instance to `user:<userId>` on the pubsub bus; idempotent.
- `unsubscribeUserChannel(userId)` — unsubscribe once the last socket for a user disconnects.

## Technical Debt

- Call sites for `publishToUser` still call it as a synchronous void function (no `await`). These must be updated in Iteration 2 to properly await the now-async function.
- `subscribeUserChannel`/`unsubscribeUserChannel` are not yet wired into the WebSocket lifecycle in `api/index.ts` — also Iteration 2.

## What Should Be Done Next
