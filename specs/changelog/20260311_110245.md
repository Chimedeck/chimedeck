# Changelog — 2026-03-11T110245Z

## Update
- `server/extensions/attachment/api/index.ts` — wired four new multipart routes into the attachment router (start, part-url, complete, abort) before the existing single-file routes.

## New
- `server/extensions/attachment/api/multipart/start.ts` — `POST /api/v1/cards/:id/attachments/multipart/start`: validates MIME type & file size, initiates S3 `CreateMultipartUpload`, creates a PENDING attachment row, returns `{ attachmentId, uploadId, key }`.
- `server/extensions/attachment/api/multipart/partUrl.ts` — `POST /api/v1/cards/:id/attachments/multipart/part-url`: validates partNumber (1–10000), verifies S3 key belongs to a PENDING attachment on the card, returns a pre-signed `UploadPart` URL (TTL 5 min).
- `server/extensions/attachment/api/multipart/complete.ts` — `POST /api/v1/cards/:id/attachments/multipart/complete`: calls `CompleteMultipartUpload` on S3, enqueues virus scan, publishes `attachment_added` WS event, returns the persisted attachment row.
- `server/extensions/attachment/api/multipart/abort.ts` — `DELETE /api/v1/cards/:id/attachments/multipart/:uploadId`: aborts the S3 multipart upload (best-effort), deletes the PENDING attachment row to prevent orphan accumulation.
- `tests/integration/attachments/multipart.test.ts` — 12 handler-level integration tests covering validation paths for all four endpoints (bad JSON, missing fields, disallowed MIME, oversized file, out-of-range partNumber, missing key query param).

## Technical Debt
- S3 success paths (card-not-found 404, attachment-not-found 404 for part-url/complete/abort) require a running LocalStack instance; not covered by the current handler-level test suite — full flow tests should be added as LocalStack integration tests in a future iteration.
