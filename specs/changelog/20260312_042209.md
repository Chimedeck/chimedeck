# Changelog — 20260312_042209

## Update
- `LogPanel/index.tsx` — now subscribes to the `automation_ran` WebSocket event via the shared socket singleton. New rows are prepended to the run log table in real time on page 1. `automations` prop added (optional, passed from `AutomationPanel`) for automation-name resolution. `prependedRuns` is cleared on page navigation to avoid duplicates.
- `AutomationPanel/index.tsx` — passes `automations` down to `LogPanel` so WS-prepended rows can show the correct automation name.
- `AutomationList.tsx` — each `AutomationRow` subscribes to `automation_ran` and maintains a local `runDelta` counter. `RunCountChip` is shown alongside the rule name (grey at 0, green when > 0).
- `ButtonsTab.tsx` — each `ButtonRow` subscribes to `automation_ran` and shows a `RunCountChip`. `runDelta` increments optimistically on every matching WS event.
- `ScheduleItem.tsx` — subscribes to `automation_ran`, shows a `RunCountChip` with the same grey/green semantics.

## New
- `src/extensions/Automation/components/LogPanel/RunCountChip.tsx` — compact per-automation run-count badge. Grey border when `count === 0`, emerald when `count > 0`. Caps display at "999+" to prevent layout overflow. Accessible `aria-label` included.
- `tests/e2e/automation/logPanel.md` — extended with new test scenarios: real-time row prepend, run-count chip state on Rules/Buttons/Schedule tabs, deduplication on reconnect.

## Technical Debt
- Run-count deltas are held only in local component state (not Redux). If the automation list is reloaded (e.g. user leaves and returns to the tab), the delta resets and the server-side `runCount` is shown instead — which is correct but may lag by up to one tick behind the live badge value.

## What Should Be Done Next
- Consider persisting `runDelta` in a Redux slice so increments survive tab switches within the same panel session.
- If `automation_ran` event payload is extended to include `automationName`, the `automations` prop dependency in `LogPanel` can be removed.
- Reconnect scenario: on WS reconnect, call `getBoardRuns` for page 1 and diff against `prependedRuns` to surface any missed events.
