# Changelog — 20260313_071013

## Update

- `server/extensions/notifications/api/index.ts` — mounted `/preferences` sub-router before existing routes.
- `server/mods/flags/defaults.ts` — added `NOTIFICATION_PREFERENCES_ENABLED: true`.
- `server/config/env.ts` — exposed `NOTIFICATION_PREFERENCES_ENABLED` env var (defaults on, disable with `NOTIFICATION_PREFERENCES_ENABLED=false`).

## New

- `server/extensions/notifications/api/preferences/get.ts` — `GET /api/v1/notifications/preferences`: returns all 4 notification types with current preferences for the authenticated user; missing rows default to both channels enabled (opt-out model).
- `server/extensions/notifications/api/preferences/update.ts` — `PATCH /api/v1/notifications/preferences`: upserts a preference row for a given type; validates type and boolean fields strictly.
- `server/extensions/notifications/api/preferences/index.ts` — thin router that checks the `NOTIFICATION_PREFERENCES_ENABLED` feature flag and delegates to GET/PATCH handlers.
- `server/extensions/notifications/mods/preferenceGuard.ts` — exports `NOTIFICATION_TYPES` constant, `NotificationType` type, and `preferenceGuard({ userId, type })` helper that looks up a preference row and falls back to `{ in_app_enabled: true, email_enabled: true }` when none exists.
- `specs/tests/notification-preferences-api.md` — Playwright MCP spec covering: default GET for new user, PATCH insert, PATCH update, invalid type/value rejections, unauthenticated 401, feature-flag-off 501, per-user isolation.

## Technical Debt

- `preferenceGuard` is imported by `get.ts` only for the `NOTIFICATION_TYPES` constant; once `dispatch.ts` is wired in (Iteration 3) it will also call the guard function itself.
