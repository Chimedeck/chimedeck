# Changelog — 20260323_120000

## Update

- `server/extensions/notifications/mods/boardPreferenceGuard.ts`: Added `selectChannels` (pure cascade helper) and `resolveNotificationChannels` (async db-backed resolver). The cascade order is: board-type-pref → user-type-pref → default (true, true). Global and board-global toggles remain the caller's responsibility.
- `server/extensions/notifications/mods/emailDispatch.ts`: Added optional `emailEnabled` parameter to `dispatchNotificationEmail`. When provided by the caller (post-`resolveNotificationChannels`), Gate 2 (user-level preference check) is bypassed, honouring the resolved board override.
- `server/extensions/notifications/mods/boardActivityDispatch.ts`: Replaced flat `preferenceGuard` call in both `handleBoardActivityNotification` and `dispatchDirectCardNotification` with `resolveNotificationChannels`. Both `inApp` and `email` channels are now resolved via the board-type → user-type → default cascade. The resolved `emailEnabled` value is forwarded to `dispatchNotificationEmail` so board-level email overrides are respected end-to-end. Removed unused `preferenceGuard` import.

## New

- `server/extensions/notifications/mods/__tests__/boardPreferenceGuard.test.ts`: Unit tests for `selectChannels` covering T1 (board override beats user master), T2 (fallback to user row), T3 (default both-on), and T6/T7 (global guard is caller responsibility). Uses pure function testing — no db mocking required.
- `server/extensions/notifications/mods/__tests__/boardActivityDispatch.test.ts`: Integration tests verifying that `resolveNotificationChannels` is called per-recipient with correct args, and that the resolved `emailEnabled` value is forwarded to `dispatchNotificationEmail`.

## Technical Debt

- The `db('notifications').insert(...)` path in `boardActivityDispatch` is not easily unit-testable due to Bun's mock-module shared registry; the dispatch tests skip the in-app insert path by using `inApp: false` in email-focused tests.
