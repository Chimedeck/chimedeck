# Changelog — 2026-03-12 12:00:00

## Update

- **`card.copy_to_board` automation action** — new action that copies the triggering card to a list on a different board. The copy includes `title`, `description`, `due_date`, and computed fractional position (`top` / `bottom`). Config fields: `targetBoardId`, `targetListId`, `position`.

- **Automation CRUD endpoints now require workspace membership** (`create.ts`, `update.ts`, `delete.ts`) — previously any authenticated user who knew a board ID could create or mutate automations for it. Now requires at least `MEMBER` role in the board's workspace.

- **Button run endpoints fixed** (`runCardButton.ts`, `runBoardButton.ts`) — both referenced a non-existent `board_members` table. Replaced with the correct `memberships` query via the board's `workspace_id`.

- **Same-board guard added to `card.move_to_list`** — target list is now verified to belong to `automation.board_id`, preventing cross-board card movements via misconfigured rules.

- **Same-board guard added to `list.move_all_cards`** — both `fromListId` and `toListId` are verified to belong to `automation.board_id`.

## New

- `server/extensions/automation/engine/actions/card/copyToBoard.ts` — `card.copy_to_board` action handler. Enforces that the **triggering actor** (`evalContext.actorId`) has `MEMBER+` workspace membership on the target board — making the action user-specific: the same rule succeeds or fails per user based on their actual board access.

- `server/extensions/board/api/workspaceBoards.ts` — `GET /api/v1/boards/:boardId/workspace/boards` endpoint returning all `ACTIVE` boards in the same workspace. Used by the automation config UI board picker. Requires workspace membership.

- `src/extensions/Automation/components/AutomationPanel/RuleBuilder/configFieldRenderer.tsx` — added `board-select` and `target-list-select` field types. Refactored `renderConfigField` into a dispatch-map pattern to stay within cognitive complexity limits.

- `src/extensions/Automation/components/AutomationPanel/RuleBuilder/ActionConfig.tsx` — now fetches workspace boards for the `board-select` picker. When `targetBoardId` changes, dynamically fetches that board's lists for the `target-list-select` field. Clears `targetListId` when `targetBoardId` is changed.

## Technical Debt

- `z.string().uuid()` is deprecated in the version of Zod used — all action handlers use it. No replacement is made here since it is a pre-existing project-wide issue.
- `randomUUID` from `'crypto'` vs `'node:crypto'` is a pre-existing lint warning across multiple files.
- `runBoardButton.ts` and `create.ts`/`update.ts` Cognitive Complexity lint violations are pre-existing.

## What Should Be Done Next

- Copy labels, members, and checklist items as optional config flags (`copyLabels`, `copyMembers`, `copyChecklists`) on `card.copy_to_board`.
- Consider a "Move card to another board" variant that removes the card from the source list after copying.
- Add an activity log entry on both the source and target boards referencing the `actorId` when a card is copied cross-board.
- Run a migration or audit to ensure all existing automation rows with `card.move_to_list` / `list.move_all_cards` configs reference lists on the same board (the new same-board guard now rejects previously-misconfigured rules that may have silently worked).
