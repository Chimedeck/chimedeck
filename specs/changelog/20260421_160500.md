## Update
- Continued medium-priority short-ID migration for board/list/label-related API paths.
- Normalized board identifiers at the list router boundary so `/api/v1/boards/:id/lists` routes accept both board UUIDs and board short IDs.
- Hardened list handlers to operate on canonical board IDs (resolved board rows) instead of reusing raw route params.

## New
- `listRouter` now resolves board identifiers through `resolveBoardId` before dispatching to list handlers.
- `handleListLists` now resolves board identifiers defensively when invoked directly.
- `handleCreateList` and `handleReorderLists` now use canonical `board.id` for list queries, inserts, and emitted events.
- Board label handlers now use canonical board IDs for DB operations; label creation also resolves incoming board identifiers defensively.

## Technical Debt
- List/checklist/label entities still do not have dedicated short-id columns, so endpoints that are directly keyed by list/checklist/label IDs remain UUID-based.
- Additional board handlers that currently rely on upstream router normalization could be made resolver-safe in-handler for stronger local guarantees.

## What Should Be Done Next
- Add integration tests for list and label endpoints using both board UUID and board short ID route inputs.
- Decide whether list/checklist/label entities should gain short-id support; if yes, add schema + resolvers and migrate list/checklist route params.
- Continue medium-priority board guest/member handler hardening where direct `boardId` lookups still rely on router-level normalization.
