# Changelog — 2026-03-10T10:54

## Update

- Auth cookies (`refresh_token`) in login, refresh, logout, and OAuth callback handlers now include the `Secure` flag (`SameSite=Strict; Secure; HttpOnly`), completing the required cookie security posture.

## New

- **`server/middlewares/csrfGuard.ts`**: New middleware that validates the `Origin` (falling back to `Referer`) header on all `POST`, `PUT`, `PATCH`, `DELETE` requests. Returns `403 csrf-origin-mismatch` if the header is present but does not start with `APP_BASE_URL`. Requests with no origin header are allowed (non-browser clients).
- **`server/config/env.ts`**: Added `APP_BASE_URL` env var (falls back to `APP_URL` for backward compatibility) used by the CSRF guard.
- **`server/index.ts`**: `csrfGuard` is now registered globally in the `fetch` handler immediately after `parseJsonBody`, covering all mutation routes.
- **`tests/e2e/csrf-guard.md`**: Playwright MCP markdown test covering correct-origin pass-through, mismatched-origin rejection (Origin and Referer), no-header allow, safe-method exemption, and cookie flag verification.

## Technical Debt

- The `Secure` cookie flag will cause the refresh token cookie to be dropped on local HTTP (`http://localhost`) in strict browser environments. Developers testing the OAuth flow locally may need to use `https://localhost` or a proxy. Consider making the `Secure` flag conditional on `NODE_ENV=production` if this causes friction.
