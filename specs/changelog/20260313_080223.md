# Changelog — 2026-03-13 (Sprint 73 Iteration 9)

## Update

- `src/extensions/Notification/components/NotificationItem.tsx`: Replaced hardcoded "mentioned you in" copy with a type-aware `buildCopy` function. Each of the 4 notification types now renders distinct copy: `card_created` → "X created Y in Board", `card_moved` → "X moved Y to List" (graceful fallback without list if absent), `card_commented` → "X commented on Y", `mention` → "X mentioned you in Y". Unread indicator moved inline next to text. Type icon replaces the unread dot: `AtSymbolIcon` (mention), `RectangleStackIcon` (card_created), `ArrowRightIcon` (card_moved), `ChatBubbleLeftEllipsisIcon` (card_commented), each with a distinct accent colour class.
- `src/extensions/Notification/api/index.ts`: Added `NotificationType` union export (`mention | card_created | card_moved | card_commented`), extended `Notification.type` to use it, extended `source_type` to include `'board_activity'`, and added optional `list_title` field for card_moved destination list.
- `src/extensions/Notification/hooks/useNotificationSync.ts`: Added explanatory comment clarifying the hook handles all notification types generically — no code logic change needed since the reducer is type-agnostic.
- `server/extensions/notifications/api/list.ts`: Added `LEFT JOIN lists ON cards.list_id = lists.id` to the query and surfaces `lists.name as list_title` in each serialised notification. This populates `list_title` when fetching card_moved notifications via the REST API.
- `server/extensions/notifications/mods/boardActivityDispatch.ts`: For `card_moved` events, the WS `notification_created` payload now includes `list_title` (destination list name sourced from `templateData.toList`). This ensures real-time panel updates show the correct "moved to X" copy without an extra round-trip.

## New

- `specs/tests/board-activity-notifications-ui.md`: Playwright MCP spec with 10 scenarios covering all notification types' icon/copy rendering, real-time WS append, mark-as-read flow, preference guard opt-out (in_app_enabled:false), API type filter, and graceful fallback when list_title is null.

## Technical Debt

- The `Notification.type` field is typed as `NotificationType | string` to remain forward-compatible with future types not yet constrained at the client layer. If the set of types is definitively closed, the `| string` escape hatch can be removed.
- `useNotificationSync.ts` and `useNotifications.ts` naming diverged (file is `useNotificationSync`); the plan referenced `useNotifications.ts`. Both refer to the same hook — consider renaming in a future refactor.
