# Sprint 09 — Real-Time Infrastructure

## Update

- `server/mods/cache/index.ts` — replaced the flat `memCache` export with a feature-flag-driven `CacheProvider` singleton; `memCache` is re-exported from the new nodeCache adapter for full backward compatibility.
- `server/index.ts` — `Bun.serve` now handles WebSocket upgrades (`/ws` path) before HTTP routing; `wsHandlers` are passed as the `websocket` config.
- `server/extensions/board/api/index.ts` — added `GET /api/v1/boards/:id/events` and `GET /api/v1/boards/:id/presence` routes.
- 15 mutation handlers across board / list / card — `console.log('[event] ...')` stubs replaced with real `writeEvent()` calls that persist to the `events` table and publish to pub/sub.

## New

- **DB migration** `db/migrations/0008_events.ts` — creates `events` (append-only log with bigint `sequence`) and `board_snapshots` tables.
- **Event store** `server/mods/events/write.ts` + `read.ts` — `writeEvent()` persists a row, triggers snapshot policy check, and publishes to pub/sub (both non-blocking). `readEventsSince()` fetches ordered events after a given sequence.
- **Snapshot system** `server/mods/events/snapshot/` — `write.ts` (upsert), `read.ts` (load snapshot + delta), `policy.ts` (trigger every 50 events).
- **Cache abstraction** `server/mods/cache/types.ts` + `adapters/nodeCache.ts` + `adapters/redis.ts` — `CacheProvider` interface, `NodeCacheAdapter` (in-process), `RedisCacheAdapter` (ioredis).
- **Pub/sub abstraction** `server/mods/pubsub/types.ts` + `index.ts` + `publisher.ts` + `subscriber.ts` + `adapters/inMemory.ts` + `adapters/redis.ts` — `PubSubProvider` interface, feature-flag-driven singleton (in-memory for dev, Redis when `USE_REDIS` flag + `REDIS_URL` env are set).
- **WebSocket realtime** `server/extensions/realtime/` — room registry (`rooms` map), subscribe/unsubscribe helpers with presence cache, broadcast, heartbeat loop (30 s ping / 60 s eviction), JWT auth verification, upgrade handler and message dispatcher.
- **REST polling** `GET /api/v1/boards/:id/events?since=` and `GET /api/v1/boards/:id/presence`.

## Technical Debt

- Snapshot materialization is minimal — the snapshot `state` is carried forward from the previous snapshot rather than being rebuilt by replaying all events. Full event-sourced materialisation should be added once domain reducers are defined.
- `workspace_members` is queried during WebSocket `subscribe` messages; a membership cache would reduce DB load under high connection churn.
- Redis pub/sub adapter uses `lazyConnect: true` but never explicitly calls `connect()` — ioredis will auto-connect on first command, but explicit connect with error handling would be safer.

## What Should Be Done Next

- **Sprint 10** — Activity log / notifications: surface `events` rows to users as a human-readable activity feed per board/card.
- Implement full snapshot materialisation by defining board-state reducers that fold domain events into a typed snapshot structure.
- Add integration tests that spin up a real SQLite database and verify the end-to-end `writeEvent → broadcast → client receives` flow.
- Expose a client-side WebSocket hook (React) that connects, authenticates, and merges server events into local state.
