# Changelog — 2026-03-13 (Sprint 72 Iteration 7)

## Update

- `server/mods/events/dispatch.ts`: Extended to call `handleBoardActivityNotification` fire-and-forget after every event persist. Handles `card.created`, `card.moved`, and `comment_added` events.
- `server/extensions/notifications/mods/createNotifications.ts`: Extended `createNotificationsForMentions` to dispatch mention email (fire-and-forget) after each in-app notification is created. Added optional `cardTitle` and `boardName` params. All callers updated to pass these values.
- `server/extensions/comment/api/create.ts`: Switched from `writeEvent` to `dispatchEvent` so the comment event flows through the full events pipeline (board activity notifications + automation).
- `server/extensions/comment/api/update.ts`: Passes `cardTitle` and `boardName` to `createNotificationsForMentions` for mention email context.
- `server/extensions/card/api/update.ts`: Passes `cardTitle` and `boardName` to `createNotificationsForMentions` for mention email context.

## New

- `server/extensions/notifications/mods/boardActivityDispatch.ts`: New module that fetches workspace members (excluding actor) and dispatches per-member email notifications via `dispatchNotificationEmail` for `card.created`, `card.moved`, and `comment_added` events. Failures are fire-and-forget (logged, never propagated).
- `tests/integration/notifications/emailNotifications.test.ts`: 13 integration tests covering flag gating, unsupported event filtering, fire-and-forget guarantee, and all 4 email template rendering scenarios. All pass.
- `specs/tests/email-notifications.md`: Playwright MCP spec with 9 end-to-end scenarios covering SES flag gating, mention email, board activity emails, actor exclusion, preference opt-out, SES failure resilience.

## Technical Debt

- Board activity dispatch only handles the email channel in this iteration. In-app channel for `boardActivityDispatch` is deferred to Sprint 73 (Iteration 8).
- Integration tests use unit-level assertions (no real DB). Full DB-backed tests with real member fetching are deferred.
- `comment_added` event type is now routed through `dispatchEvent`. Automation triggers for comments still use `card.comment_added` (different type); this inconsistency pre-existed and was not introduced here.

- Sprint 73 (Iteration 9): Client notification panel icons/copy for new types + WS handler update.
