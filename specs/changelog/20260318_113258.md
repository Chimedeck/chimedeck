# Sprint 87 – Client Optimistic Delete + Rollback

**Date:** 2026-03-18
**Iteration:** 2 of 11

---

## Update

- `src/extensions/Board/api.ts`: Added `BoardDeleteConflictError` interface that types the 409 Conflict response payload (`{ name: 'delete-requires-confirmation'; data: { listCount, cardCount } }`).
- `src/extensions/Board/containers/BoardListPage/BoardListPage.duck.ts`: Added `deleteSnapshot: Board[] | null` to state; wired `deleteBoardOptimisticThunk` via `extraReducers` (pending → snapshot + remove, fulfilled → clear snapshot, rejected → rollback). Re-exports `deleteBoardOptimisticThunk` as the canonical delete action for the page.
- `src/extensions/Board/containers/BoardListPage/BoardListPage.tsx`: Updated `handleDelete` to dispatch `deleteBoardOptimisticThunk` instead of the legacy non-optimistic `deleteBoardThunk`.
- `specs/tests/board-delete-auto-refresh.md`: Expanded BDAR-UI-01 (immediate removal) and BDAR-UI-02 (rollback on failure) from placeholder stubs to full Playwright MCP scenarios.

---

## New

- `src/extensions/Board/slices/boardsSlice.ts`: New slice file that exports `deleteBoardOptimisticThunk` — a `createAppAsyncThunk` that performs the HTTP DELETE and returns the `boardId` on success. The boards-array state lives in `boardListPage`; this slice drives the optimistic flow via RTK's cross-slice extraReducers pattern.

---

## Technical Debt

- The pre-existing thunks in `BoardListPage.duck.ts` (fetch, create, archive, etc.) use `extra.api` without casting `extra` from `unknown`; this produces pre-existing TS errors. These are not introduced by this iteration and should be cleaned up in a dedicated typing hardening pass.
- `deleteBoardThunk` (the old non-optimistic thunk) is still exported from `BoardListPage.duck.ts` but is no longer dispatched by the UI. It should be removed once all callers are migrated.
