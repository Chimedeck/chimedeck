## Update
- Added support for body-based batch custom-field-value fetching to avoid oversized URLs and 431 errors when many card IDs are requested.

## New
- Server now accepts `POST /api/v1/boards/:boardId/custom-field-values` with payload `{ cardIds: string[] }`.
- Legacy `GET /api/v1/boards/:boardId/custom-field-values?cardIds=...` remains supported for backward compatibility.
- Client batch custom field fetch now uses POST by default and falls back to GET for older servers that return 404/405.
- Added e2e API test coverage for POST batch endpoint in `tests/e2e/custom-field-values-api.spec.ts`.

## Technical Debt
- If boards become extremely large, request body size could still be high; consider chunking `cardIds` client-side in the future.

## What Should Be Done Next
- Add server-side limit + validation error for excessively large `cardIds` arrays (e.g. > 10k) with clear error message.
- Consider deprecating GET batch route after all clients are on POST.
