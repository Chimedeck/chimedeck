## Update
- Replaced card tile drag hook from `useSortable` to `useDraggable` in `CardItem` to remove per-card sortable subscription overhead in the drag hot path.
- Removed card-level `SortableContext` usage in `ListColumn`; card insertion preview continues to be driven by pointer/midpoint placeholder logic in `BoardCanvas`.

## New
- Card tiles now use draggable transform translation from DnD core (`CSS.Translate`) instead of sortable transform state.

## Technical Debt
- `BoardCanvas` drag handlers remain complex and still contain mixed responsibilities (pointer tracking, collision filtering, placeholder state, commit fallback logic).
- Keyboard drag behavior for card-level sortable semantics should be re-verified because card tiles are now draggable-only.

## What Should Be Done Next
- Re-profile the same 300+ card drag scenario and compare `useSortable` self-time and total `CardItem` time against the previous trace.
- If INP is still above target, isolate expensive card content blocks (`CardPluginBadges`, `CardCustomFieldBadges`, avatars, due-date formatting) behind drag-time guards for non-active cards.
- Add a focused interaction test to verify same-list and cross-list insertion indices after the draggable migration.
