# Changelog — 20260324_090356

## Update
- `server/extensions/healthCheck/api/index.ts`: Registered two new probe endpoints and fixed route ordering so `probe-all` is matched before the `/:id` wildcard.

## New
- `server/extensions/healthCheck/mods/rateLimiter.ts`: In-memory rate limiter (5 s window). Exports `checkRateLimit(key)` and `retryAfterMs(key)`. Keyed by healthCheckId or boardId. Stateless across process restarts (by design).
- `server/extensions/healthCheck/api/probe.ts`: `POST /api/v1/boards/:boardId/health-checks/:id/probe` — runs an on-demand HTTP probe for a single health check, applies per-check rate limiting (1 req / 5 s), returns `{ data: PersistedProbeResult }`.
- `server/extensions/healthCheck/api/probeAll.ts`: `POST /api/v1/boards/:boardId/health-checks/probe-all` — probes all active health checks on a board concurrently, applies per-board rate limiting (1 req / 5 s), silently skips individual checks that are within their own 5 s window, returns `{ data: PersistedProbeResult[] }`.

## Technical Debt
- Rate limiter state is in-memory only; horizontally scaled deployments will not share rate limit state. A Redis-backed limiter would be needed for multi-process/multi-pod setups.
- probe-all silently skips rate-limited individual checks — the caller cannot distinguish "skipped" from "not yet probed". A future iteration could include a `skipped` flag in the response.
