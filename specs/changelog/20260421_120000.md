# Update
- Started incremental board hydration implementation to reduce first-load latency on large boards.
- Updated board read flow to support partial initial card payload per list when feature flag `INCREMENTAL_BOARD_HYDRATION` is enabled.
- Added pagination support to list-card endpoint for background batch fetches.
- Extended board client/store plumbing to track per-list hydration progress and merge batched card payloads.
- Added per-column loading indicator UI while additional cards are being hydrated.

# New
- Board API now accepts `initialCardsPerList` on board read and returns optional `includes.card_hydration` metadata.
- List cards API now supports `limit` and `offset`, and returns pagination metadata (`total`, `nextOffset`, `hasMore`) when paginated.
- New board-store async thunk for list-level card batch hydration.

# Technical Debt
- Background hydration scheduling is currently implemented in board canvas render layer; may be extracted into a dedicated hydration coordinator hook for cleaner separation.
- Existing lint debt in board container/canvas files remains and was not broadly refactored in this iteration.
- Current first-slice uses list-scoped pagination endpoint reuse; a dedicated board-level hydration endpoint may still be desirable for centralized orchestration and metrics.

# What Should Be Done Next
- Add integration tests for partial board payload + paginated list-card batches under feature-flag on/off paths.
- Add E2E scenario for large board verifying first screenful render and eventual full hydration.
- Add telemetry around time-to-first-cards and time-to-full-hydration before production canary rollout.
- Evaluate moving hydration scheduler to a standalone hook with visibility observer and bounded concurrency controls.
